# Council — 4声評議会

曖昧な意思決定のために4つのアドバイザーを召集する：
- **Architect**（本人の視点）— 正確性・保守性・長期的影響
- **Skeptic** サブエージェント — 前提への疑問・単純化・仮定の破壊
- **Pragmatist** サブエージェント — 実装速度・ユーザー影響・運用上の現実
- **Critic** サブエージェント — エッジケース・ダウンサイドリスク・失敗モード

**意思決定の曖昧さ**を扱うもの。コードレビュー・実装計画・アーキテクチャ設計には使わない。

## 使う場面

以下のときに使う：
- 複数の有力な選択肢があり明確な勝者がない
- トレードオフを明示的に洗い出したい
- 会話のアンカリング（最初に提示された案に引きずられる）を避けたい
- go / no-go の判断に異論を挟みたい

**flowsurface での典型例：**
- ワークスペース分割 vs モノリス維持（新機能をどのクレートに置くか）
- iced の `Task` vs `Command` の移行タイミング
- 取引所アダプタを共通 trait で抽象化する vs 個別実装のまま維持する
- リプレイエンジンをリファクタリングして今リリースする vs 機能追加を優先する
- E2E テストを HTTP API 経由に統一する vs ユニットテストで代替する

## 使わない場面

| council の代わりに | 使うもの |
|---|---|
| コードが正しいか検証したい | `verification-loop` |
| 実装を手順に分解したい | 計画書を `docs/plan/` に作成 |
| コードのバグ・安全性レビュー | `rust-reviewer` |
| 明確な事実への答え | 直接答える |
| 実装タスクの実行 | そのままやる |

## 役割

| 声 | 視点 |
|---|---|
| **Architect** | 正確性・保守性・Rust の型安全性・長期的影響 |
| **Skeptic** | 前提への疑問・最もシンプルな代替案・「本当に必要か」 |
| **Pragmatist** | 実装コスト・ユーザーへの即効性・今すぐ動くこと |
| **Critic** | エッジケース・失敗モード・導入後に後悔するリスク |

3つの外部の声は**フレッシュなサブエージェント**として起動する。アンカリング防止のため、進行中の会話履歴は渡さない。

## ワークフロー

### 1. 本当の問いを絞り出す

決定を1つの明確なプロンプトに落とす：
- 何を決めるのか？
- 制約条件は何か（性能・保守コスト・リリース期限）？
- 成功の定義は何か？

曖昧な場合は評議会を召集する前に1つだけ確認する。

### 2. 最小限のコンテキストを集める

コードベース固有の決定なら：
- 関連ファイル・スニペット・issue テキストを収集
- コンパクトに保つ（関係する部分だけ）

戦略的な決定なら：
- リポジトリのスニペットは答えを変えない限り省略

### 3. Architect の立場を先に固める

外部の声を読む前に書き留める：
- 自分の初期ポジション
- その根拠となる3つの理由
- 自分が選ぶパスの最大のリスク

これを先に書くことで、外部の声をただ反映するだけの合成を防ぐ。

### 4. 3つの独立した声を並列起動する

各サブエージェントに渡すもの：
- 決定の問い
- 必要なら最小限のコンテキスト
- 厳格な役割定義
- 会話履歴は渡さない

プロンプトの形：

```text
あなたは4声評議会の [役割] です。

問い：
[決定の問い]

コンテキスト：
[関連するスニペットまたは制約のみ]

以下の形式で回答してください：
1. ポジション — 1〜2文
2. 根拠 — 簡潔な箇条書き3点
3. リスク — あなたの推奨の最大のリスク
4. 意外な観点 — 他の声が見落とすかもしれないこと

直接的に。曖昧にしない。300字以内。
```

役割の強調：
- **Skeptic**: 前提への疑問、最もシンプルな代替案の提示、仮定の破壊
- **Pragmatist**: 速度・シンプルさ・現実的な実行を最適化
- **Critic**: ダウンサイドリスク・エッジケース・計画が失敗する理由

### 5. バイアスガードで合成する

ルール：
- 外部の見解を理由なく却下しない
- 外部の声で推奨が変わった場合は明示する
- 最も強い異論は却下する場合でも必ず含める
- 2つの声が自分の初期ポジションに反対したら、それを真剣なシグナルとして扱う

### 6. コンパクトな判定を出す

```markdown
## Council: [短い決定タイトル]

**Architect:** [1〜2文のポジション]
[理由1行]

**Skeptic:** [1〜2文のポジション]
[理由1行]

**Pragmatist:** [1〜2文のポジション]
[理由1行]

**Critic:** [1〜2文のポジション]
[理由1行]

### 判定
- **合意点:** [一致しているところ]
- **最強の異論:** [最も重要な不一致]
- **前提チェック:** [Skeptic が問い自体に疑問を呈したか]
- **推奨:** [合成されたパス]
```

スマートフォンの画面で読めるくらいコンパクトに。

## 判定後の処理

判定が何かを実際に変えるなら：
- 重大な技術的意思決定 → `architecture-decision-records` で ADR に記録する
- 実装に影響するなら → `docs/plan/` の計画書を更新する
- GitHub issue に反映すべきなら → 直接更新する

判定ごとにメモを書き散らさない。判定が何かを変えた場合だけ記録する。

## 複数ラウンド

デフォルトは1ラウンド。

追加ラウンドが必要なら：
- 新しい問いを絞る
- 前回の判定は本当に必要な場合のみ含める
- Skeptic はできるだけクリーンに保つ（アンカリング防止）

## アンチパターン

- コードレビューに council を使う
- 実装作業に council を使う
- 会話の全履歴をサブエージェントに渡す
- 最終判定で異論を隠す
- 重要でない決定のたびに記録を残す
